Database indexing is a mechanism whereby the retrieval of rows from a table is optimized through the utilization of auxiliary data structures. The most commonly utilized data structure in the context of indexing is the B-tree, which is a self-balancing tree data structure that maintains sorted data and allows searches, sequential access, insertions, and deletions in logarithmic time. When a query is executed against a table that has an index defined on the relevant column or columns, the query planner may elect to utilize the index in order to avoid a full table scan.

It should be noted that indexes are not free. The maintenance of an index incurs overhead on write operations, because every insertion, update, or deletion that affects an indexed column necessitates a corresponding modification of the index structure. Therefore, the decision regarding whether to create an index involves a tradeoff between read performance and write performance, and this tradeoff must be evaluated in the context of the specific workload characteristics of the application in question.

Furthermore, there exists the concept of a composite index, which is an index that is defined on multiple columns. The ordering of columns within a composite index is significant due to the fact that the index can only be utilized for queries that filter on a prefix of the indexed columns. This is sometimes referred to as the leftmost prefix rule, and a failure to understand it frequently results in indexes that are not utilized by the queries that the developer intended them to optimize.
