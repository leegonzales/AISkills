# Permission Model â€” Session Stamping

FleetOps write verbs (`journal add`, `journal update`) are attributable. Every write must be traceable to a specific shell session with a specific agent identity. This file explains why, how, and what goes wrong when the stamp is missing.

## Why

Without session stamping, a write to `fleet.db` has no provenance. Two parallel shells writing "as Adama" would produce indistinguishable rows. Debugging a bad stanza ("who wrote this? from where?") becomes impossible. Audit trails on doctrine-sensitive writes become useless.

Stamping gives every row:

- `session_id` â€” unique per shell, generated by `fleetops session-stamp`
- `spawn_type` â€” how the shell was started (`cic`, `heartbeat`, `mail`, `dream`, `manual`)
- `agent` â€” the agent identity claimed by the shell

The DB enforces these on write. A rejected write is a feature, not a bug.

## How â€” Setup

At the start of any shell that will write to `fleet.db`:

```bash
eval "$(fleetops session-stamp cic)"
export FLEETOPS_AGENT=<your agent name, exact casing>
```

- `fleetops session-stamp` prints two export lines. The `eval` applies them to the current shell.
- Spawn-type arg is one of `cic | heartbeat | mail | dream | manual`. Default if omitted: `cic`.
- `FLEETOPS_AGENT` is **not** set by `session-stamp` â€” you set it explicitly. This is deliberate: one session-stamped shell can be reused across multiple agent identities if the operator is testing.

Verify:
```bash
echo "$FLEETOPS_SESSION_ID $FLEETOPS_SPAWN_TYPE $FLEETOPS_AGENT"
```

## How â€” Read vs Write

- **Read verbs** (`show-state`, `tail-events`, `wakes`, `search-knowledge`, `journal tail`, `journal show`, `journal grep`, `journal render`, `list-agents`, `doctor`) do not require stamping. Read whenever.
- **Write verbs** (`journal add`, `journal update`) require `FLEETOPS_SESSION_ID` and `FLEETOPS_AGENT`.
- `fleetops harvest` and `fleetops rebuild-state` are operator/daemon verbs with their own service-account identity; they don't use the interactive stamp.

## Daemon-spawned sessions

When the `servitor` daemon spawns a wake session (heartbeat, mail, dream), it stamps the child shell automatically via the embedded `CLAUDE_SERVITOR.md` protocol. You should not need to stamp manually inside a daemon-spawned wake â€” the environment already carries `FLEETOPS_SESSION_ID` and `FLEETOPS_SPAWN_TYPE`, and your agent persona sets `FLEETOPS_AGENT` on the first write.

Check:
```bash
env | grep ^FLEETOPS_
```

If those variables are absent in a daemon-spawned session, something upstream is wrong â€” surface it to Adama. Do not bypass by manually stamping; the daemon expects to own that state.

## Failure modes

### "rejected: no session_id"

```
fleetops: journal add: no FLEETOPS_SESSION_ID in environment
```

Cause: the shell was not stamped.

Fix:
```bash
eval "$(fleetops session-stamp cic)"
```

### "rejected: no agent identity"

```
fleetops: journal add: no FLEETOPS_AGENT and no --agent flag supplied
```

Cause: the shell was stamped but has no agent attached.

Fix:
```bash
export FLEETOPS_AGENT=Adama    # or pass --agent Adama on the command
```

### "rejected: agent mismatch"

```
fleetops: journal add: --agent "Daystrom" does not match stamped session agent "Adama"
```

Cause: trying to write as an agent other than the one the session is stamped for, without elevation.

Fix: either restamp the shell (`eval "$(fleetops session-stamp cic)"` + `export FLEETOPS_AGENT=<target>`) or, if you are running a cross-agent migration (the bulk-import kata is one), use `--agent` explicitly and accept that all rows will carry your stamped session â€” that's the correct behavior for an operator-supervised migration.

### "rejected: unknown agent"

```
fleetops: journal add: agent "adama" not in registered list (check fleetops list-agents)
```

Cause: agent-name casing or spelling mismatch. Agents are case-sensitive and match `~/.servitor/fleet-roster.md`.

Fix: `fleetops list-agents` â†’ copy the exact name.

### Silent mis-attribution

If you stamp once at the top of a long-running interactive session and then write as multiple different agents via `--agent`, all those rows carry the same `session_id`. That is correct if you're operator-supervising (you're one human at one keyboard). If you want crisp attribution, re-stamp per agent.

## Summary

1. Stamp every shell before writing: `eval "$(fleetops session-stamp cic)"; export FLEETOPS_AGENT=<name>`.
2. Read without stamping, write with stamping.
3. Daemon-spawned sessions are pre-stamped; don't touch.
4. Rejections are signal â€” read the stderr message and fix the cause, don't bypass.
